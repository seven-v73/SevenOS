# SevenOS Hyprland Lua Engine

SevenOS does not replace Hyprland with Lua directly. Hyprland consumes its own configuration language, so the stable migration path is:

```text
Lua intent -> generated Hyprland conf -> safe source include -> Hyprland reload
```

This keeps SevenOS crash-proof while moving toward a programmable desktop.

## Goals

- Audit the current Hyprland configuration.
- Separate static configuration from dynamic SevenOS behavior.
- Generate profile-aware Hyprland snippets from Lua.
- Keep `custom.conf` and the existing `.conf` files as fallback.
- Prepare an event/runtime layer for future automation.

## Structure

```text
hyprland/lua/
  init.lua
  core/

```

## Generated File

Lua writes:

```text
hyprland/conf/sevenos-lua-generated.conf
```

The live session receives:

```text
~/.config/hypr/conf/sevenos-lua-generated.conf
```

The file is sourced by `hyprland/hyprland.conf`, but it is safe by default. If Lua fails, the last valid generated file or the static fallback remains active.

## Phase 0 Output

The audit command creates a machine-readable map:

```text
hyprland/lua/config_map.json
~/.config/sevenos/hypr-lua/config_map.json
```

It includes bind counts, window rule counts, animation authority, source includes, dynamic surfaces, external SevenOS script references and semantic duplicate checks.

## Phase 2 Output

The plan command creates a profile runtime contract:

```text
hyprland/lua/profile_runtime.json
~/.config/sevenos/hypr-lua/profile_runtime.json
```

It exposes the selected mini OS intent:

- runtime signals
- workspace intent
- future event actions
- generated environment hints
- generated window rules

This is the bridge between LAPA profiles and Hyprland behavior. The active profile can be generated normally, while another profile can be previewed safely:

```bash
seven hypr lua plan pulse --json
seven hypr lua generate pulse
```

Only `apply` copies the generated layer into the live Hyprland config directory.

## Phase 3 Output

The Lua engine now emits:

- shared SevenOS keybinds from `hyprland/lua/rules/keybinds.lua`
- shared SevenOS animation presets from `hyprland/lua/rules/animations.lua`
- shared SevenOS window rules from `hyprland/lua/rules/windows.lua`
- shared SevenOS workspace rules
- profile-specific environment hints
- profile-specific layout tuning
- profile-specific window routing

Profile activation also refreshes the Lua-generated layer:

```text
seven profile activate forge
  -> writes profile state
  -> applies runtime/isolation policy
  -> runs seven hypr lua apply forge
  -> restarts Waybar/profile visuals
```

This makes Lua part of the real SevenOS profile lifecycle while keeping the static config as a fallback.

## Phase 4 Output

Lua is now the authority for dynamic desktop behavior:

- `hyprland.conf` keeps stable compositor foundations.
- common keybinds are generated by Lua.
- common window rules are generated by Lua.
- common animations and profile motion overrides are generated by Lua.
- profile layout snippets are generated by Lua.
- the audit reports duplicate static authority.

SevenOS also ships a lightweight Hyprland event bridge:

```bash
seven-hypr-lua-events status --json
seven-hypr-lua-events once
seven-hypr-lua-events watch
```

The event bridge listens to Hyprland's socket when available and writes events to:

```text
~/.config/sevenos/hypr-lua/events.jsonl
```

This gives SevenOS a safe path toward `onWindowOpen`, `onWorkspaceChange` and `onAppFocus` behavior without making the compositor depend on an experimental daemon.

When `socat` is available, the bridge listens to Hyprland's event socket. If `socat` is missing, it automatically uses `nc -U` or `ncat --unixsock` when present. Only if no Unix socket stream helper exists does SevenOS fall back to lightweight polling.

The bridge also writes the last interpreted state to:

```text
~/.config/sevenos/hypr-lua/current.json
```

That file exposes the active profile, observed context and latest safe action, so Spotlight, Waybar, Profile Center and future AI layout logic can react without parsing raw Hyprland events.

## Migration Rule

Never delete the static Hyprland config during early phases. The Lua engine is additive until it proves it can reproduce every major behavior.
